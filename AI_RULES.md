# AI 开发规则

## 1. 总原则

本项目是包网平台底座，技术路线为：

- B 端后台：RuoYi-Vue-Plus
- C 端 H5 / 官网 / 活动页 / PWA：Vue3 + Vite
- 玩家 App：Flutter
- 自研游戏 / 活动小游戏：预留 Cocos Creator 接入
- 核心后端：Spring Boot / Java
- 钱包体系：多币种钱包中心

AI 开发必须遵守边界优先、账务安全、配置驱动、可审计、可测试。

## 2. 禁止事项

- 禁止修改 RuoYi-Vue-Plus 框架核心包，除非任务明确要求。
- 禁止绕过权限、租户、数据权限机制。
- 禁止业务模块直接修改钱包余额。
- 禁止在 Controller 中写复杂业务逻辑。
- 禁止直接拼接 SQL。
- 禁止新增未经确认的第三方依赖。
- 禁止大范围重构已有代码。
- 禁止把渠道开关、币种能力、地区规则写死在前端。
- 禁止在日志、文档、测试数据中写入真实密钥、真实用户信息、支付凭证、KYC 文件地址。
- 禁止点击或自动执行真实充值、出款、兑换、人工调账、封禁、解封等生产级操作。

## 3. 模块边界

| 模块 | AI 可做 | AI 不可做 |
| --- | --- | --- |
| tenant-center | 租户、品牌、域名、主题配置 | 绕过租户隔离 |
| channel-center | 渠道功能开关 | 在前端写死渠道规则 |
| member-center | 会员资料、状态、标签 | 直接改钱包余额 |
| wallet-center | 币种、账户、流水、冻结、结算 | 被其他模块绕过 |
| game-center | 游戏列表、启动、投注回调、派彩回调 | 直接操作余额表 |
| promotion-center | 活动、任务、奖励发放 | 绕过钱包发奖励 |
| payment-center | 支付订单、回调、对账 | 在未验签时入账 |
| redemption-center | 兑换申请、审核、出款状态 | 审核成功前直接扣减余额 |
| risk-center | 地区、KYC、黑名单、风控规则 | 只在前端做风控 |
| audit-center | 敏感操作日志 | 删除或覆盖审计记录 |

## 4. 后端编码规则

- Controller 只负责参数接收、权限注解、返回结果。
- Service 负责业务流程和事务边界。
- Mapper 只负责数据访问。
- 钱包变更必须调用 `wallet-center` 服务接口。
- 每个账务类接口必须有业务单号，并保证幂等。
- 数据库变更必须提供 SQL 或 migration。
- 新接口必须说明权限标识、租户隔离、幂等规则。
- 真金、兑换、冻结、人工调账必须写审计日志。

## 5. 前端编码规则

- H5、App、后台只能通过 API 获取币种、渠道、功能开关。
- 不允许前端自行判断钱包核心规则。
- 不允许页面直接写死地区合规逻辑。
- 活动页和品牌样式必须走租户 / 品牌配置。
- 复用统一 API 类型和错误码。
- 钱包金额展示必须使用后端返回的精度和展示规则。

## 6. 测试规则

涉及以下功能时必须提供测试或可验证脚本：

- 钱包入账、扣账、冻结、解冻、结算。
- 支付回调。
- 游戏投注回调。
- 游戏派彩回调。
- 活动奖励发放。
- 兑换审核。
- 人工调账。
- 租户隔离。
- 渠道开关。

## 7. 提交规则

每次提交只处理一个主题。

提交信息建议格式：

```text
feat(wallet): add multi-currency account draft
docs(plan): add phase 1 implementation plan
test(wallet): cover idempotent debit
fix(redemption): prevent duplicate settlement
```

提交前必须说明：

- 改了哪些文件。
- 新增或修改了哪些接口。
- 是否涉及数据库变更。
- 是否涉及权限菜单。
- 是否涉及钱包账务。
- 如何验证。
