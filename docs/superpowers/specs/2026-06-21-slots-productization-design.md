# Tang Luck Slots Productization Design

## Objective

Upgrade Tang Luck from a wallet, campaign, purchase, KYC, and redemption workflow into a game-led product experience. The C-side first impression must feel like a casino slots product, while the B-side must support the operational configuration, auditability, and review workflows required for a formal production product.

The implementation is intentionally split into three independently verifiable stages:

1. B1: Slots core game loop
2. B2: Campaign and task linkage
3. B3: Production operations engagement

## Product Principles

- The first visible C-side experience should be a game lobby, not a collection of operational links.
- GC is the only playable currency in this phase.
- SC is visible only as a future eligibility/configuration switch and is not usable for wagering in this phase.
- Slot results are generated by the backend so rounds, wallet effects, and audit records are testable and reviewable.
- The feature is a sandbox productionization step, not certified real-money RNG or gambling infrastructure.
- Every C-side capability must have corresponding B-side configuration, query, and audit support.

## B1: Slots Core Game Loop

### User Experience

C-side adds or upgrades the game experience:

- Game lobby presents real Slots game cards as the primary product surface.
- A game detail page is available at a route such as `/app/slots/:gameCode`.
- The Slots page presents a five-reel layout, symbol strip, spin button, GC balance, bet controls, and recent round history.
- A user can choose a GC bet amount, spin once, receive a backend-generated result, and see:
  - round ID
  - bet amount
  - win multiplier
  - payout amount
  - updated GC balance
  - latest history row
- Recent game history shows the last rounds with bet, payout, status, and time.

### Backend Behavior

New domain tables:

- `slot_games`
  - `game_code`
  - `title`
  - `status`
  - `sort_order`
  - `min_bet_gc`
  - `max_bet_gc`
  - `default_bet_gc`
  - `rtp_profile`
  - `sc_enabled`
  - `created_at`
  - `updated_at`

- `slot_rounds`
  - `round_id`
  - `user_id`
  - `game_code`
  - `currency`
  - `bet_amount`
  - `result_symbols_json`
  - `multiplier`
  - `payout_amount`
  - `status`
  - `wallet_debit_ledger_id`
  - `wallet_credit_ledger_id`
  - `created_at`

New C-side APIs:

- `GET /api/v1/slots/games`
  - Returns active game cards for the C-side lobby.

- `POST /api/v1/slots/{gameCode}/spin`
  - Validates authenticated user, active game, allowed bet range, and sufficient GC.
  - Debits GC through `wallet_ledger`.
  - Generates backend-controlled five-reel result.
  - Credits GC payout when payout is greater than zero.
  - Persists `slot_rounds`.
  - Returns round result, wallet delta, and updated balance.

- `GET /api/v1/slots/rounds`
  - Returns the authenticated user's recent slot rounds.

### B-Side Operations

New admin pages:

- `/admin/games`
  - List games.
  - Activate or pause games.
  - Configure sort order and GC bet range.
  - Display SC enabled flag as reserved, default disabled.

- `/admin/game-rounds`
  - Query rounds by user, game, currency, status, and time.
  - Show bet, payout, multiplier, result symbols, linked wallet ledgers, and created time.

Admin operations write `audit_logs` for game status and configuration changes.

### B1 Boundaries

- GC only.
- No SC wagering.
- No certified RNG claims.
- No provider integration.
- No complex payline editor in the first version.

## B2: Campaign And Task Linkage

### User Experience

B2 connects Slots play to operational campaigns and daily tasks:

- Game lobby shows active campaign panels such as Daily Spin, cumulative bet, cumulative win, and consecutive login.
- Slots page shows task progress near the game surface:
  - today's spins
  - cumulative GC bet
  - cumulative GC won
- Activity center becomes a task board with these states:
  - in progress
  - claimable
  - completed
  - expired
- Claiming a completed task grants GC through the wallet ledger.
- Lobby and activity center update after spin progress changes.

### Backend Behavior

Campaign task model supports target types:

- `spin_count`
- `bet_amount`
- `win_amount`
- `daily_login`
- `consecutive_days`

New or extended persistence:

- `campaign_tasks`
  - campaign-level task configuration
  - target type and target value
  - reward GC amount
  - start and end time
  - region eligibility
  - risk action

- `campaign_task_progress`
  - user progress per task
  - current value
  - status
  - claim timestamp

Slots spin success updates relevant task progress. Reward claiming is idempotent and writes wallet ledger entries.

### B-Side Operations

`/admin/campaigns` expands to support task campaign configuration:

- target type
- target value
- reward GC
- start and end time
- eligible regions
- risk action
- rules version
- budget or grant cap

Publish validation checks:

- rules version exists
- reward is configured
- region eligibility exists
- budget or grant cap exists
- legal approval exists when required

Campaign list and detail show:

- participants
- completions
- claim count
- total GC granted

All create, publish, pause, and configuration changes write `audit_logs`.

### B2 Boundaries

- GC rewards only.
- No SC betting tasks.
- No activity popup, inbox, red dot, or task chain yet.
- No email, SMS, or push notification.

## B3: Production Operations Engagement

### User Experience

B3 adds production-grade engagement surfaces:

- Activity popup
  - shown on lobby entry for the main active campaign
  - supports "do not show today"
  - includes title, reward, countdown, and CTA

- Red dots and badges
  - bottom navigation and activity entry show claimable count
  - game cards can show limited-time campaign badges

- Reward inbox
  - shows system rewards, campaign rewards, manual grants, and compensation
  - supports unread, claimable, claimed, and expired states

- Limited-time countdown
  - shown in lobby and activity center
  - campaigns change state after expiry

- Task chains
  - supports ordered steps such as login, spin ten times, bet 5,000 GC, then claim final reward

### Backend Behavior

New persistence:

- `reward_inbox`
  - user ID
  - reward type
  - status
  - reward payload
  - source campaign or manual operation
  - expiry time

- `campaign_popup_rules`
  - campaign
  - display window
  - frequency control
  - CTA target

- `campaign_task_steps`
  - ordered task chain steps
  - unlock requirements
  - reward payload

New C-side APIs:

- `GET /api/v1/player/activity-summary`
- `GET /api/v1/player/notifications`
- `POST /api/v1/player/notifications/{id}/claim`

Reward claiming is idempotent and links to wallet ledger records.

### B-Side Operations

New or expanded admin pages:

- `/admin/notifications`
  - query user inbox
  - issue manual compensation
  - expire or cancel pending rewards

- `/admin/activity-dashboard`
  - activity exposure count
  - task completions
  - reward claims
  - total GC granted
  - abnormal user count

- `/admin/campaigns`
  - popup rules
  - task chain configuration
  - campaign engagement metrics

All manual grants, cancellations, and expiry operations write `audit_logs`.

### B3 Boundaries

- GC rewards only.
- No A/B testing.
- No push notification, SMS, or email.
- No external ad attribution integration.
- Keep attribution fields available for future expansion.

## Data Flow

### Spin Flow

1. User opens a Slots game.
2. Frontend loads game config, wallet summary, and recent rounds.
3. User selects GC bet and submits spin.
4. Backend validates bet range and balance.
5. Backend writes GC debit ledger.
6. Backend generates five-reel result.
7. Backend writes payout credit ledger when payout is greater than zero.
8. Backend persists `slot_rounds`.
9. Backend updates campaign task progress when B2 is enabled.
10. Frontend renders reel result, payout, wallet delta, and round history.

### Task Claim Flow

1. User completes task progress through spin or login behavior.
2. Backend marks task as claimable.
3. Frontend shows claimable badge and task status.
4. User claims reward.
5. Backend checks idempotency and task status.
6. Backend writes GC reward ledger.
7. Backend marks progress as claimed.
8. Frontend updates wallet, task state, and badges.

### Inbox Claim Flow

1. Reward inbox item is created by campaign completion, system action, or admin compensation.
2. User opens reward inbox.
3. User claims reward.
4. Backend checks status, expiry, and idempotency.
5. Backend writes wallet ledger.
6. Backend marks item as claimed.

## Error Handling

- Insufficient GC returns a business error and does not create a round.
- Inactive game returns a game unavailable error.
- Invalid bet range returns a validation error.
- Duplicate spin idempotency key returns the original round.
- Wallet debit success and payout failure must not leave an unclosed round; implementation should use a transaction.
- Expired campaign task cannot be claimed unless the campaign explicitly allows grace claim.
- Already claimed reward returns the existing claim state.

## Testing Strategy

### Backend

- Slot spin debits GC and records a round.
- Winning spin credits payout and links wallet ledger IDs.
- Insufficient GC does not create round or ledger.
- Inactive game blocks spin.
- Admin game config changes write audit logs.
- Task progress updates after spin.
- Task reward claim is idempotent.
- Inbox reward claim is idempotent.

### Frontend

- Lobby renders game cards before secondary operational links.
- Slots page renders five reels, bet controls, spin result, wallet delta, and history.
- Insufficient balance disables or blocks spin with clear message.
- Activity center renders in-progress, claimable, completed, and expired tasks.
- Inbox renders claimable, claimed, expired, and empty states.
- Admin games and game rounds pages render tables, filters, loading, empty, error, and action states.

### Browser Verification

- Desktop C-side lobby and Slots page.
- Mobile C-side lobby and Slots page.
- Desktop B-side games, game rounds, campaigns, activity dashboard, and notifications pages.
- English default and Chinese browser language checks.

## Acceptance Criteria

- A user can complete a GC Slots round end to end.
- Wallet balances and ledger records match the round result.
- Admin can see the game configuration and round record.
- Activity progress changes after Slots play in B2.
- Claiming task rewards writes wallet ledger and prevents duplicate reward claims.
- Reward inbox in B3 supports claimable, claimed, expired, and manual grant scenarios.
- All high-risk B-side changes have audit logs.
- Tests, build, and Playwright pass for each stage before moving to the next stage.
