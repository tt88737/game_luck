# Tang Luck 最终交付包索引

## 1. 文档定位

本索引用于承接上级目录中的 `TangLuck可运营可评审可验收最终PRD.md`。主 PRD 负责统一范围、路线、合规原则和验收口径；本文件夹中的专项文档负责给对应团队执行。

建议先阅读 `00-交付包总入口.md`，再按本文索引进入各专项文档。

## 3. 推荐执行顺序

| 顺序 | 动作 |
| --- | --- |
| 1 | 用 `00-交付包总入口.md` 说明交付包结构和阅读顺序 |
| 2 | 用主 PRD 完成客户和内部评审 |
| 3 | 用 `TangLuck高保真原型.html` 对齐 C 端和后台最终呈现 |
| 4 | 冻结 P0-A 范围和上线阻断条件 |
| 5 | 后端拆 `OpenAPI` 和数据库设计 |
| 6 | 产品/运营根据活动配置参数表冻结 P0-A 奖励、预算、SC 策略 |
| 7 | 产品/UI 根据页面原型说明补 UI 细节和文案 |
| 8 | QA 基于测试矩阵和验收检查清单出测试计划 |
| 9 | 项目经理用 15 天客户演示脚本准备评审路径 |
| 10 | 法务、支付、App 审核材料同步准备 |
| 11 | 按 15 天冲刺执行表进入开发 |

## 4. 评审建议

| 评审对象 | 优先材料 | 输出 |
| --- | --- | --- |
| 客户决策层 | 总入口、高保真原型、15 天冲刺表、15 天后路线 | 是否认可交付范围和路线 |
| 产品/运营 | 页面原型说明、高保真原型、P0-A 活动配置参数表 | 活动配置、奖励策略、运营节奏 |
| 研发 | OpenAPI 草案、数据库设计、P0-A 活动配置参数表、15 天冲刺表 | 接口、表结构、任务拆分 |
| QA | QA 测试矩阵、验收检查清单、原型、客户演示脚本 | 测试计划和验收报告模板 |
| 法务/合规 | 法务确认包、App 上架审核包、支付商准入包 | 外部确认清单 |

## 5. 使用注意事项

1. 本交付包是产品、研发、运营和评审材料，不构成美国法律意见。
2. 涉及 SC、AMOE、兑换、州级开放、支付、App 上架的事项，必须以美国律师、支付商、Apple/Google 的正式确认为准。
3. P0-A 15 天版本重点是合规活动运营最小闭环，不建议把真实支付、真实兑换、复杂 VIP、自动风控塞进第一期。
4. 高保真原型用于评审和对齐，不代表最终视觉设计稿；正式开发前 UI 仍需结合品牌和技术栈细化。
